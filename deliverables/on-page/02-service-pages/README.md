# Service Pages

**Process ID:** 02
**Category:** On-Page SEO
**Fulfillment Connection:** Onsite Core Content Optimization
**Cadence:** Onboarding, launch rebuilds, new service launches, and as-needed optimization updates
**Automation Readiness Score:** 7/10 - Highly automatable
**Status:** Documented

---

## Purpose

Service pages turn the approved keyword map into pages that can rank, convert, and support the rest of the SEO system. A strong service page should make it obvious what the client does, where they do it, why the service matters, what the process looks like, and what the visitor should do next.

This process receives input from Keyword Research and feeds:

- Service Area Pages
- Blog Content
- Internal Linking
- Alt Text on Images
- Title Tags and Meta Descriptions
- H1 Optimization
- Schema Markup
- Google Search Console review
- Internal Monthly Scorecard
- Client Monthly Report

## Current State

The current baseline documentation identifies service pages as part of Onsite Core Content Optimization, but the placeholder did not define a complete operating standard. The automation docs already describe URL architecture and website content briefs, but the team still needs a practical SOP for turning those briefs into publishable service pages.

Without this SOP, service pages can drift into thin copy, generic headings, repeated stock sections, weak internal links, missing schema, or content that ranks for a keyword but does not answer the homeowner's real buying question.

## Target State

Every priority service should have a clear page target, either as a core service page or an individual service page. The page should be grounded in the approved keyword map, written for the client's actual market, and published with the required SEO elements in place.

The final page must include:

- Approved primary keyword and secondary keywords
- Search intent and funnel stage
- Correct URL recommendation
- Clear title tag, meta description, and H1
- Useful body copy with service-specific proof
- Client-specific differentiators
- Local relevance when appropriate
- FAQ section
- Internal links to related services, service areas, blogs, and conversion pages
- Image guidance and alt text
- Schema recommendation
- Human approval notes
- Publish or optimization status

## Workflow

### 1. Confirm the Page Assignment

Start with the approved keyword map and confirm the page's role.

Classify the page as one of:

- Core service page
- Individual service page
- Existing service page optimization
- New service page
- Service page consolidation
- Service page removal or redirect

Document:

- Target URL
- Existing URL if different
- Primary keyword
- Secondary keywords
- Service category
- Funnel stage
- Priority tier
- Search intent
- Assigned owner

If the keyword map does not clearly assign the page, return to Process 01 before drafting.

### 2. Validate the Service and Offer

Confirm the client actually wants leads for the service.

Check:

- The service is currently offered
- The service is profitable enough to prioritize
- The client has capacity to fulfill it
- The service is available in the target market
- The client has enough proof, photos, or detail to support the page
- The service is not seasonal unless the page explains seasonality
- The page will not compete with another existing page

Do not publish a page for a service the client does not want to sell. Rankings for bad-fit work are not wins.

### 3. Review the Existing Page or SERP

For an existing page, audit:

- URL
- Title tag
- Meta description
- H1
- Word count and section structure
- Above-the-fold clarity
- Service specificity
- Local relevance
- Internal links
- Images and alt text
- FAQs
- Schema
- Conversion path
- Duplicate or cannibalized content
- Indexing/canonical status

For a new page, review the local SERP and top competitors to understand:

- Common page sections
- Questions competitors answer
- Service proof competitors use
- Local language or neighborhood references
- Gaps the client's page can answer better

Competitor review is for gap detection. Do not copy wording, layouts, or claims.

### 4. Build the Page Brief

Create a brief before drafting. The brief should be short enough to use and complete enough to prevent generic copy.

Required brief fields:

| Field | Description |
|---|---|
| Client | Client name/domain. |
| Page Type | Core service, individual service, or optimization. |
| Target URL | Final URL path. |
| Current URL | Existing URL when optimizing. |
| Primary Keyword | Main approved target. |
| Secondary Keywords | Supporting terms and synonyms. |
| Search Intent | Transactional, commercial, informational, or mixed. |
| Funnel Stage | Usually BOFU for service pages. |
| Service Summary | Plain-English description of the offer. |
| Audience | Homeowner, commercial property manager, HOA, builder, or other. |
| Pain Points | What the buyer is trying to fix. |
| Proof Points | Reviews, years in business, photos, project examples, process, equipment, certifications. |
| Required Sections | H2 plan and conversion sections. |
| Internal Links | Links to include on the page. |
| Schema | Recommended schema types. |
| Human Notes | Service limitations, claims to avoid, approval notes. |

### 5. Define the Page Structure

Use structure that matches search intent. Avoid making every service page the same block with a different keyword pasted in.

Minimum service page structure:

1. Hero or opening section
2. Service explanation
3. Problems the service solves
4. Process or what to expect
5. Service-specific benefits
6. Client differentiators
7. Related services
8. Service area/local relevance when appropriate
9. FAQs
10. Conversion CTA

Recommended H2 patterns:

| Page Need | Example H2 Direction |
|---|---|
| Buyer education | What Is [Service]? |
| Problem framing | When Your Property Needs [Service] |
| Process clarity | How Our [Service] Process Works |
| Trust | Why Homeowners Choose [Client] |
| Local fit | [Service] for [Market] Properties |
| Conversion | Schedule [Service] |
| Support content | Frequently Asked Questions About [Service] |

### 6. Draft or Rewrite the Page

The page should answer the buyer's actual questions before it tries to sound impressive.

Writing standards:

- Use the primary keyword naturally in the title tag, H1, intro, and at least one supporting section.
- Use secondary keywords where they fit naturally.
- Explain the service in plain language.
- Include client-specific process details.
- Include local context only when it is true and useful.
- Use real differentiators instead of generic claims.
- Avoid thin "we provide quality service" filler.
- Avoid keyword stuffing.
- Avoid unsupported superlatives like "best" unless there is proof.
- Avoid fake guarantees, prices, timelines, or certifications.
- Keep the CTA clear and consistent with the client's site.

If a page cannot be made specific because client information is missing, document the gap and request the detail before final approval.

### 7. Optimize Metadata

Every service page needs:

- Title tag
- Meta description
- H1
- Clean URL slug
- Canonical URL
- Open Graph title and description when supported

Metadata rules:

| Element | Standard |
|---|---|
| Title tag | Primary keyword plus brand or market context, usually under 60 characters when practical. |
| Meta description | Service, market, differentiator, and CTA, usually 140-160 characters when practical. |
| H1 | Clear page topic, not a vague marketing slogan. |
| Slug | Short, readable, service-focused. |
| Canonical | Self-referencing unless consolidation requires another canonical. |

Use Process 07 for deeper metadata auditing, but do not publish a service page with blank or obviously weak metadata.

### 8. Add FAQs

Include FAQs when they help the buyer and fit the page.

FAQ standards:

- 3-5 questions per service page is usually enough.
- Questions should answer real buying objections.
- Answers should be concise and service-specific.
- Avoid repeating the same FAQ set across every service.
- Avoid fake pricing answers if pricing varies.
- Add FAQPage schema only when the FAQs are visible on the page.

Good FAQ themes:

- Timeline
- Seasonality
- Cost variables
- Property fit
- Process
- Maintenance
- Service limits
- What happens during an estimate

### 9. Plan Internal Links

Each service page should connect to the rest of the site.

Minimum link set:

- Link to the parent/core service page when the page is an individual service.
- Link to related individual services.
- Link to relevant service area pages.
- Link to one or more supporting blog posts when available.
- Link to contact/estimate page.
- Link from related pages back to the service page when appropriate.

Anchor text should be descriptive and natural. Do not use exact-match anchors everywhere.

### 10. Add Images and Alt Text Guidance

For each page, identify the images needed:

- Hero/service image
- Process or crew image
- Before/after or finished project image when available
- Supporting visual for FAQ/process sections if the design needs it

Alt text should describe the image and service context. Do not stuff keywords into image alt text.

Good alt text example:

`Crew installing mulch around a residential planting bed in Princeton, NJ`

Weak alt text examples:

- `image`
- `landscaping`
- `best landscaping Princeton NJ landscaping company`

Use Process 06 for full image-alt auditing.

### 11. Add Schema Recommendations

Most service pages should be considered for:

- LocalBusiness or Organization schema at the site level
- Service schema when supported by the CMS/schema plugin
- FAQPage schema when visible FAQs exist
- BreadcrumbList schema

Schema should match visible page content. Do not add FAQ schema for hidden or nonexistent FAQs. Do not invent prices, ratings, aggregate reviews, or service areas.

### 12. Human Review and Approval

Human approval is required before publishing or major optimization updates.

The reviewer checks:

- Service accuracy
- Client voice and claims
- Service limits
- Local details
- Differentiators
- CTA
- Images
- Any pricing, guarantee, or compliance-sensitive claims
- Whether the page should publish now or wait for more detail

No page is complete until a human approves it or leaves revision notes.

### 13. Publish or Prepare CMS Update

Publishing steps depend on the site stack.

For WordPress/Rank Math:

- Update page content.
- Update Rank Math title and description.
- Confirm index/canonical settings.
- Add or confirm schema.
- Add FAQs if supported by the page builder.
- Add images and alt text.
- Clear cache.
- Verify the live page.

For static/repo sites:

- Update the route/page file.
- Update any image manifest or asset metadata.
- Run local checks.
- Commit and deploy.
- Verify the production URL.

### 14. Post-Publish QA

After publishing, verify:

- Page returns HTTP 200.
- Page is indexable unless intentionally noindexed.
- Title tag is present.
- Meta description is present.
- H1 is present and unique enough.
- Canonical is correct.
- Internal links work.
- Images load.
- Alt text is present where needed.
- CTA works.
- FAQ/schema output is valid if used.
- Mobile render is clean.
- Page is included in sitemap when appropriate.
- GSC indexing request is submitted when needed.

### 15. Monthly or As-Needed Optimization

Use GSC and rankings to improve existing service pages.

Watch for:

- High impressions with low CTR
- Positions 4-20 for important service terms
- Keywords ranking on the wrong page
- Cannibalization between service pages and blogs
- Pages with weak engagement or no conversions
- New service demand
- Competitor pages adding stronger coverage

Optimization can include metadata rewrites, new sections, FAQ expansion, internal links, image improvements, schema cleanup, or page consolidation.

## Automation Plan

### Koga/Kai Can Automate

- Pull approved keyword map data.
- Match service keywords to existing URLs.
- Generate page briefs.
- Audit existing page metadata, H1s, links, schema, and word count.
- Pull GSC page/query data.
- Review SERP competitors and summarize gaps.
- Draft or rewrite service page copy.
- Draft title tags and meta descriptions.
- Generate FAQ candidates.
- Recommend internal links.
- Generate alt text recommendations.
- Prepare schema recommendations.
- Build QA checklists.
- Identify pages that may need consolidation or redirects.

### Human Must Handle

- Confirm the service is real and actively sold.
- Approve client-specific claims.
- Provide missing service details.
- Approve pricing, warranty, guarantee, or certification language.
- Approve final page copy before publishing.
- Make page-builder edits when automation cannot safely update the page.
- Decide on consolidation when two pages compete.
- Validate live page appearance and conversion fit when design changes are involved.

## Completion Criteria

This process is complete when:

1. The service page brief exists.
2. The target URL and page role are approved.
3. The content is drafted or optimized.
4. Metadata, H1, FAQs, internal links, image guidance, and schema notes are complete.
5. Human approval is documented.
6. The page is published or the publish-ready update packet is staged.
7. Live QA confirms the page works and is indexable when appropriate.
8. The page can feed internal linking, metadata, schema, GSC, and reporting workflows without rework.

## Common Mistakes

- Creating service pages from a raw service list without keyword approval.
- Publishing a page for a service the client does not want to sell.
- Reusing the same generic copy structure across every service.
- Targeting the same keyword on multiple pages without a consolidation decision.
- Adding fake local details.
- Writing for Google before answering the homeowner's actual question.
- Skipping metadata because the body copy is done.
- Adding FAQ schema for FAQs that are not visible.
- Using exact-match internal links everywhere.
- Forgetting to clear cache and verify the live page.
- Calling the page complete before human approval.
